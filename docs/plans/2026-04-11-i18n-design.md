# S3C Manager I18n Design

**Date:** 2026-04-11

**Scope:** Whole product i18n unification for frontend UI, backend API messages, install flow, and demo content.

## Goal

Unify the product around a single, consistent bilingual model with canonical locales `cs` and `en`, without adding locale prefixes to URLs. The active language should come from the signed-in user's preference when available, otherwise from a locale cookie, otherwise from the system/browser language, and finally fall back to `cs`.

## Current Problems

- Frontend has no real i18n layer; UI strings are hardcoded across `frontend/app/` and feature components.
- Backend route messages are also hardcoded, mostly in Czech, with no shared translation model.
- Locale handling is inconsistent: HTML uses `cs`, user data uses legacy `cz`, and formatters/sorts often hardcode `'cs'`.
- Install, auth, admin, and profile flows mix Czech and English in the same screen.
- Demo/seed content is product-facing and currently exists only in one language.

## Decision

Implement a shared in-repo i18n layer with:

- canonical locales: `cs`, `en`
- legacy input normalization: `cz`, `cze`, `cs-CZ` -> `cs`; `en`, `en-US`, `en-GB` -> `en`
- no locale prefixes in routes
- locale resolution order:
  1. authenticated user `preferred_lang`
  2. `sc_locale` cookie
  3. `Accept-Language` / system locale
  4. fallback `cs`

## Architecture

### Shared i18n module

Create a shared module, for example `shared/i18n/`, containing:

- `locales.ts`
  - canonical locale constants
  - legacy locale normalization
  - request/system locale resolution helpers
- `messages/cs.ts`
- `messages/en.ts`
- `catalog.ts`
  - typed message catalog access
  - fallback logic
- `translate.ts`
  - `t(locale, key, params)` for backend
  - provider-facing helpers for frontend

This module becomes the single source of truth for message keys used by both frontend and backend.

### Frontend integration

Frontend gets a lightweight i18n layer:

- `I18nProvider`
- `useT()`
- `useLocale()`
- locale-aware formatting helpers for date/time and string compare

The resolved locale is injected at the app shell/layout level so SSR and client renders use the same locale. The current `html lang="cs"` becomes dynamic.

### Backend integration

Backend gets:

- `resolveRequestLocale(req)`
- `t(locale, key, params)`
- middleware/helper support so route handlers can translate messages without duplicating logic

The response schema does not change. Only message contents become locale-aware.

## Language Persistence Model

### Canonical values

Only `cs` and `en` are stored going forward.

### Read path

- Existing stored `cz` must continue working.
- Any legacy or malformed value is normalized before use.

### Write path

- User preference writes only canonical `cs` or `en`.
- Locale cookie also stores canonical value.

### Migration

Add a DB migration that updates existing `platform.users.preferred_lang` values from `cz` to `cs` and leaves `en` unchanged. Unknown values should be normalized to `cs`.

## Frontend Migration Strategy

Migrate in this order:

1. app layout, shell, auth bootstrap
2. navigation and shared chrome
3. login
4. install flow
5. user profile and preferences
6. administration pages
7. major catalogue/C3 screens
8. remaining feature pages and states

Rules:

- No new hardcoded UI text outside catalogs
- `Intl.DateTimeFormat` and `localeCompare` use active locale
- profile language switch updates DB preference and locale cookie

## Backend Migration Strategy

Migrate these areas first:

1. auth middleware and auth routes
2. install routes and install service
3. admin routes
4. import/export routes
5. taxonomy and common validation paths

Rules:

- Error and success messages go through `t()`
- Locale is resolved consistently for authenticated and unauthenticated requests
- Legacy `preferred_lang: 'cz'` is exposed back to clients as canonical `cs`

## Demo and Seed Content

System/UI messages must be fully bilingual.

User/business data should remain data, not be auto-translated at runtime. For demo seed content, provide separate `cs` and `en` seed text variants because that content is part of product presentation and onboarding.

## Fallback Behavior

Fallback order for message lookup:

1. active locale key
2. `cs` key
3. visible placeholder including message key

This avoids mixed-language crashes while still making missing keys obvious in development and tests.

## Testing Strategy

### Unit tests

- locale normalization
- locale resolution from user/cookie/header
- message fallback behavior
- backend translation helper behavior

### Backend route tests

- auth/install/admin responses respect locale
- preference update stores canonical locale
- `cz` input is normalized to `cs`

### Frontend tests

- SSR/client render in `cs` and `en`
- login/install/profile flows render translated copy
- language preference change persists across refresh
- logout keeps cookie/system fallback behavior

### Regression checks

- targeted grep/check for hardcoded strings in key app-shell/auth/install/admin zones
- build + middleware tests + selected Playwright smoke tests in both locales

## Risks

- SSR/CSR locale mismatch if locale resolution is split across multiple code paths
- regression in auth bootstrap if locale loading introduces extra async boundaries
- partial migration leaving mixed `cz`/`cs` semantics in data and tests

## Risk Mitigation

- resolve locale once per request/bootstrap path
- canonicalize locale immediately at every boundary
- migrate shell/login/install/auth first
- keep the provider/helper layer intentionally thin

## Outcome

After the refactor:

- the product consistently supports `cs` and `en`
- locale is stable across SSR, client navigation, login/logout, and profile changes
- frontend and backend share one message system
- legacy `cz` values stop leaking into runtime behavior
