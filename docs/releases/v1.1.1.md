# V1.1.1 Release Notes

## Topic

Service Governance Cockpit, Help Center, and Dynamic C3 Capability Maps

## Short Description

S3C Manager v1.1.1 turns the v1.1 release line into a broader production-ready governance release. It adds a full Help experience, dynamic C3 capability map creation, a redesigned Operations cockpit, improved C3 dashboard drilldowns, cleaner production documentation, runtime version alignment, and verified lab load testing up to 100 parallel users.

The release focuses on helping organizations understand their service portfolio, detect missing ownership and metadata, identify C3 mapping gaps, expose duplicated contractual coverage, and review overloaded service managers or owners from one operational cockpit.

## Executive Summary

- Added a complete Help website entry point and aligned Help documentation with the application structure.
- Added contextual service governance views for risk, offering evidence, ownership, contract overlap, renewal risk, and C3 mapping.
- Added dynamic C3 capability map support so administrators can create new baselines such as `Spiral 99`.
- Added C3 dashboard parent-capability drilldown for better explanation of aggregated chart data.
- Removed the legacy FMN Air C2 frontend page from the production route list.
- Updated application versioning and deployment metadata to `1.1.1`.
- Cleaned production documentation and moved development-only planning artifacts out of the production docs tree.
- Validated middleware, frontend build, Docker rebuild, route smoke, and lab concurrency tests.

## What Changed

### Help Center and Documentation

- Added a full Help website entry at `/help`; current builds serve Czech help from `/help-cs` and English help from `/help-en`.
- Sidebar Help now links to the complete Help website instead of a popup-only experience.
- Removed the redundant `Help for this page` sidebar button from the production navigation.
- Help content was aligned with the documented information architecture and the main application areas.
- The Help structure covers user, content manager, and administrator responsibilities.
- Documentation now explains:
  - how to create a service catalogue record
  - how service records relate to C3 entities and capability maps
  - how results, completeness, readiness, and dashboard indicators are derived
  - how administrators manage installation, import, users, modules, and operating procedures
  - how content ownership and documentation governance should be maintained
- Production `docs/` was cleaned so release, user, admin, installation, operations, C3, import, and upgrade documentation remain visible.
- Development-only planning artifacts were moved to the ignored `dev/docs/` area.

### C3 Capability Maps

- C3 Capability Builder can create new capability map baselines such as `Spiral 99`.
- Custom maps are available through dynamic routes such as `/c3/capability-map-spiral99`.
- The frontend dynamic C3 route resolves custom `capability-map-spiral<number>` pages without requiring a dedicated static page per spiral.
- Backend taxonomy endpoints now support arbitrary normalized spiral codes such as `Spiral_99`.
- Capability map titles are stored per spiral and default to `C3 Taxonomy Catalogue — Baseline <number>`.
- Builder items are saved under the selected spiral instead of being limited to Spiral 6 or Spiral 7.
- The capability map switcher can show additional spiral baselines returned from the API.
- A regression test was added for custom capability map routes and custom spiral item storage.

### C3 Dashboard

- The `Top Parent Capabilities` chart now supports click drilldown.
- Clicking a parent capability opens an explanation dialog with the number of related items.
- Root capabilities are explained as top-level capabilities without a parent.
- The dialog links to the filtered C3 catalogue so users can inspect the underlying C3 records.
- C3 list filtering now handles root capability parent filters consistently.
- The `Needs mapping` and governance-related C3 links continue to point to C3 entity detail pages where the finding is about a C3 capability rather than a service edit action.

### Operations Cockpit

- Operations was reorganized into focused tabs:
  - `Health`
  - `Governance`
  - `Offering evidence`
  - `Owners`
  - `C3 mapping`
- The `Health` tab focuses on readiness, incomplete metadata, lifecycle cleanup, and headline operational indicators.
- The `Governance` tab groups Service Risk Radar, Owner Load Monitor, Contract Overlap, Governance Advisor, and renewal risk.
- The `Offering evidence` tab groups evidence coverage, missing evidence, contract overlap, and renewal evidence.
- The `Owners` tab separates missing owner assignment from owner overload signals.
- The `C3 mapping` tab groups mapping coverage, C3 gaps, risk signals, and advisor recommendations.
- Additional progress rows and coverage indicators were added where they help users compare service quality or mapping completeness.
- Owner Load Monitor links now navigate to the owner load detail view, where assignments and role usage can be inspected.

### Governance and Service Gap Detection

- Added service governance data structures and views for contract governance and governance reporting.
- Added governance route/repository tests covering the new cockpit endpoints.
- Added support for identifying:
  - incomplete metadata
  - missing owners
  - owner overload
  - offering evidence gaps
  - contract overlap
  - renewal risk
  - C3 mapping gaps
  - service risk findings
  - governance advisor recommendations
- The cockpit is designed to support management reviews, portfolio cleanup, and consolidation decisions.

### Lab Load Test

The application was tested in a local lab against the containerized stack with authenticated parallel users hitting catalogue, operations, service, and C3 read endpoints.

| Scenario | Duration | Parallel users | Requests | Errors | p95 latency | Result |
|---|---:|---:|---:|---:|---:|---|
| Baseline concurrency | 30 s | 30 | 1,438 | 0 | 35.5 ms | Stable |
| High concurrency | 30 s | 100 | 2,441 | 0 | 38 ms | Stable |

These results are lab evidence, not a production SLA. Production capacity still needs validation on target hardware, database storage, reverse proxy, TLS, real data volume, and observability settings.

## Upgrade Notes

- Set deployment image/application version to `1.1.1`.
- Rebuild and redeploy the app container.
- Keep the existing PostgreSQL data volume and run normal schema initialization/migration startup.
- Verify `/api/health/live` and `/api/health/ready`.
- Open `/help`, `/help-cs`, `/help-en`, `/operations`, `/c3/dashboard`, and `/admin/c3-capability-builder`.
- If using the C3 module, create or verify capability map baselines from the builder.
- If using service governance features, verify owner, contract, offering evidence, and C3 mapping source data after import.

## Release Acceptance Focus

The release should be accepted when the following checks pass in the target environment:

- installation status is ready
- first admin account exists or SSO is configured
- main navigation renders without missing routes
- Help website opens from the sidebar
- Operations tabs render without client-side errors
- C3 dashboard renders and parent capability drilldown opens
- custom capability map routes work for at least one non-default spiral
- import and catalogue pages remain reachable
- governance cockpit data loads or shows empty states instead of errors
- backup and restore procedure is documented for the deployment
